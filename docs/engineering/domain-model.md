# Domain Model

이 문서는 V1 도메인 규칙을 정리합니다. 구현 과정에서 실제 entity와 API 계약에 맞춰 갱신합니다.

## Core Concepts

### User

- OAuth provider 계정으로 식별됩니다.
- 로그인 후 사용할 수 있는 장부를 가집니다.
- 사용 가능한 장부가 없으면 기본 개인 장부가 생성됩니다.
- Auth Foundation 구현에서는 `provider`, `providerUserId`, `email`, `nickname`, `lastUsedLedgerId`를 저장합니다.

### Ledger

- 개인 장부 또는 공동 장부입니다.
- 사용자는 여러 장부에 속할 수 있습니다.
- 사용자는 마지막으로 사용한 장부를 가질 수 있습니다.
- `name`, `type`, `ownerId`, 보관 여부(`archived`), 반복 거래 집계 마감일(`recurringSummaryClosingDay`)을 저장합니다.
- 반복 거래 집계 마감일은 1~31일이며, 해당 일의 다음 날부터 다음 마감일까지를 한 기간으로 계산합니다. `31`은 달력 월(1일~말일)을 뜻합니다.
- 보관된 장부는 장부 목록과 최근 사용 장부 선택 대상에서 제외됩니다.

### Ledger Member

- 장부와 사용자의 관계입니다.
- 공동 장부에서는 결제자, 멤버별 지출, 초대 권한 판단의 기준이 됩니다.
- 같은 사용자와 장부 조합은 한 번만 저장됩니다.
- Auth Foundation 구현에서는 `OWNER`, `MEMBER` role을 구분합니다.
- 공동 장부 소유자는 이름 변경, 보관, 멤버 내보내기를 수행할 수 있고 일반 멤버는 스스로 탈퇴할 수 있습니다.

### Ledger Month

- 장부의 월 단위 예산 운영 상태입니다.
- 월 예산, 마감 상태, 멤버별 할당을 묶고 개인 장부에서는 카테고리별 예산도 관리합니다.
- 공동 장부는 월 총 예산과 멤버별 할당만 사용하며 카테고리 예산과 고정비 예산 기본값을 사용하지 않습니다.
- Budget/Dashboard 구현에서는 `budgetMonth`를 `YYYY-MM` 문자열로 저장합니다.
- 같은 장부 안에서 같은 `budgetMonth`는 하나만 존재합니다.
- 마감된 월은 예산 설정과 거래 생성·수정·삭제, 반복 거래 생성을 수행할 수 없고 reopen 후 다시 수정할 수 있습니다.

### Category Budget

- 특정 `Ledger Month`와 장부 카테고리의 월 예산 금액입니다.
- 금액은 원화 정수 `Long`으로 저장하며 음수일 수 없습니다.
- 카테고리는 해당 장부에 속해야 합니다.
- 개인 장부에서만 사용합니다.

### Member Allocation

- 특정 `Ledger Month`에서 장부 멤버에게 배정한 예산 금액입니다.
- 금액은 원화 정수 `Long`으로 저장하며 음수일 수 없습니다.
- 할당 대상 사용자는 해당 장부의 멤버여야 합니다.

### Category

- 장부별 거래에 직접 선택하는 세부 분류입니다.
- 모든 카테고리는 같은 type의 통계 대분류 하나에 속합니다. 통계와 대시보드는 세부 카테고리 대신 대분류 기준으로 합산합니다.
- 기본 대분류는 `식비`, `교통`, `생활`, `수입`이며, 기본 세부 카테고리 `식비`, `카페`, `교통`, `생활`, `급여`를 생성합니다. `식비`와 `카페`는 `식비` 대분류에 속합니다.
- 사용자는 장부별 커스텀 대분류와 세부 카테고리를 만들 수 있으며, 세부 카테고리를 만들 때 대분류를 선택해야 합니다.
- 사용되지 않은 세부 카테고리는 삭제할 수 있습니다. 거래, 월 예산, 고정비 예산 또는 반복 거래 템플릿에 연결된 카테고리는 과거 내역과 계획을 보존하기 위해 삭제할 수 없습니다.
- 카테고리와 대분류는 `EXPENSE`, `INCOME` type을 가지며, 같은 장부 안에서 이름은 각각 중복될 수 없습니다.

### Fixed Budget Template

- 장부에서 매월 확보해야 하는 고정비 예산입니다.
- 이름, 지출 카테고리, 금액, 사용 여부를 가집니다. 금액은 원화 정수 `Long`이며 양수여야 합니다.
- 활성 고정비의 합계는 월 예산 설정에 표시됩니다. 아직 저장하지 않은 월 예산을 처음 조회하면 같은 카테고리의 예산 기본값으로 합산되어 미리 채워집니다.
- 고정비는 예산 계획이며 실제 거래를 생성하지 않습니다. 실제 반복 지출은 `Recurring Transaction Template`으로 관리합니다.

### Transaction

- 장부에 속한 수입 또는 지출 기록입니다.
- 금액, 일자, 카테고리, 메모, 결제자, 결제수단 정보를 가집니다.
- 월별 거래 목록, 대시보드, 통계 계산의 기준입니다.
- Transaction Core 구현에서는 금액을 원화 정수 `Long`으로 저장합니다.
- 거래 type과 카테고리 type은 일치해야 합니다.
- 결제자는 해당 장부의 멤버여야 하며, 요청에서 생략하면 현재 사용자로 저장합니다.
- 현금(`CASH`) 또는 등록 카드(`CARD`)를 결제수단으로 선택합니다. 카드는 지출 거래에서만 선택할 수 있습니다.
- 카드 지출은 2~60개월 할부로 등록할 수 있습니다. 총 결제 금액은 회차별로 나누며, 나머지 원 단위는 앞선 회차에 1원씩 배분합니다.
- 할부 거래는 같은 할부 계획 id와 회차/전체 회차 정보를 가지며, 시작일을 기준으로 매달 한 건씩 생성합니다. 한 회차라도 마감 월에 속하면 전체 등록을 거부합니다.

### Card

- 카드는 장부별로 등록하며 이름과 결제금액 확정일(매월 1~31일)을 가집니다.
- 카드 거래는 해당 장부에 등록된 카드만 선택할 수 있습니다.
- 카드의 다음 청구 예상 금액은 직전 확정일 다음 날부터 다음 확정일까지의 카드 지출을 합산해 계산합니다. 31일은 해당 월의 마지막 날로 보정합니다.
- 거래가 연결된 카드는 과거 거래 보존을 위해 삭제할 수 없습니다.

### Transaction Import Preview

- OCR 또는 직접 입력 텍스트에서 거래 후보를 만드는 임시 결과입니다.
- preview는 실제 `Transaction`을 저장하지 않습니다.
- 후보는 금액, 날짜, type, 카테고리 힌트, 메모, 원문 줄, confidence를 가집니다.
- 후보를 확정 저장할 때는 일반 거래 생성 API 규칙을 따릅니다.
- 사용자는 후보의 type, 금액, 날짜, 카테고리, 메모를 확인하고 수정한 뒤 저장합니다.

### Settlement Payment

- 공동 장부 월 정산에서 멤버 간 실제 송금 내역을 기록합니다.
- 월 예산의 멤버별 할당 비율과 멤버별 실제 지출을 이용해 받을 금액과 보낼 금액을 계산합니다.
- 할당 합계가 0이면 멤버 수로 균등 분담합니다.
- 일부 금액만 기록할 수 있으며, 기록 취소 시 남은 정산 금액이 다시 계산됩니다.

### User Notification

- 사용자별 알림과 읽음 상태를 저장합니다.
- 직접 초대, 월 마감, 예산 초과 등의 제품 이벤트가 알림을 생성합니다.
- 사용자는 개별 알림 또는 전체 알림을 읽음 처리할 수 있습니다.

### Recurring Transaction Template

- 반복적으로 생성될 거래의 템플릿입니다.
- 중복 생성을 방지하는 generation 기록이 필요합니다.
- Recurring Transaction 구현에서는 `WEEKLY`, `MONTHLY` frequency를 지원합니다.
- 템플릿은 장부, 결제자, 카테고리, 거래 type, 금액, 메모, 시작일, 다음 생성 예정일, 종료일, 일시정지 여부를 가집니다.
- 생성 대상 여부는 `paused = false`이고 `nextDueDate <= asOf`이며, 종료일이 없거나 `nextDueDate <= endDate`일 때 판단합니다.
- 반복 거래 생성은 실제 `Transaction`과 `RecurringTransactionGeneration` 기록을 함께 만들고, `(template_id, generated_date)`로 중복 생성을 방지합니다.
- 시작일이 오늘 또는 과거면 시작일 발생분을 즉시 생성하고, 미래 시작일이면 서버 작업이 발생일에 자동 생성합니다.
- 템플릿을 삭제하면 미래 자동 생성과 generation 기록만 제거하며, 이미 생성된 `Transaction`은 보존합니다.
- 생성 후 `nextDueDate`는 frequency에 따라 다음 주 또는 다음 달로 전진합니다.

### Invitation

- 공동 장부 참여 요청입니다.
- 직접 초대와 링크 초대를 지원합니다.
- pending, accepted, rejected, cancelled, expired 같은 상태를 구분합니다.
- Invitation 구현에서는 `DIRECT`, `LINK` type과 `PENDING`, `ACCEPTED`, `REJECTED`, `CANCELLED`, `EXPIRED` status를 사용합니다.
- 직접 초대는 invitee user를 가지며, 초대받은 사용자만 수락/거절할 수 있습니다.
- 링크 초대는 token을 가지며, 현재 구현은 수락되면 `ACCEPTED`가 되는 단일 사용 링크입니다.
- 공동 장부 `OWNER`만 초대 생성, 초대 가능 사용자 조회, 초대 목록 조회, 초대 취소를 할 수 있습니다.
- 초대 수락 시 사용자는 해당 장부의 `MEMBER`가 되고, 마지막 사용 장부가 해당 장부로 변경됩니다.

## Key Rules

- 모든 사용자는 최소 1개의 기본 개인 장부를 가져야 합니다.
- 최근 사용 장부에 접근할 수 없으면 기본 개인 장부로 fallback합니다.
- 거래 생성/수정은 최근 사용 장부 선택 상태를 임의로 바꾸지 않습니다.
- 개인/공동 장부는 UI에서 하나의 목록으로 표시합니다.
- API response는 JPA entity를 직접 노출하지 않습니다.
- 돈은 부동소수점으로 다루지 않습니다.
- 월 기준은 `YYYY-MM` 형식으로 표현합니다.
- 대시보드는 월을 생략하면 서버 clock의 현재 월을 사용하고, `budgetMonth`를 전달하면 해당 월을 조회합니다. 실제 지출 집계는 `EXPENSE` 거래만 합산하며, 남은 예산에는 아직 생성되지 않은 활성 지출 정기 거래의 해당 월 예정액도 차감합니다. 생성된 정기 거래는 실제 지출에만 포함해 중복 차감하지 않습니다.
- 월별 통계는 요청한 `from`부터 `to`까지의 모든 월을 포함하고, 거래가 없는 월은 0원으로 반환하며 카테고리별 지출도 함께 제공합니다.
- 카테고리별 지출 통계는 통계 대분류별로 합산합니다.
- 초대는 공동 장부에만 생성할 수 있습니다.
- 같은 장부와 invitee 조합에 만료되지 않은 직접 초대가 이미 있으면 새 직접 초대를 만들 수 없습니다.
- 반복 거래 템플릿의 카테고리 type은 거래 type과 일치해야 합니다.
- 반복 거래 템플릿의 결제자는 해당 장부의 멤버여야 합니다.
- 정산 완료 여부는 계산 결과와 실제 송금 기록으로 판단하며 화면에서 임의로 완료 처리하지 않습니다.
