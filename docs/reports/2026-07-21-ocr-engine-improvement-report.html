거래 이미지 OCR 엔진 개선 보고서

브라우저 OCR을 백엔드 Native Tesseract 파이프라인으로 이전하고, 어두운 화면 전처리, 여러 이미지 업로드, 희소 레이아웃 인식, 거래 의미 파싱과 회귀 검증까지 진행한 작업 기록입니다.

작성일 2026-07-21 대상 우리로그 로컬 개발 환경 상태 구현·로컬 검증 완료

1. 요약

오늘 작업의 핵심은 OCR 엔진을 단순히 “더 선명하게 읽게” 만든 것이 아니라, 이미지 인식과 거래 의미 해석을 분리한 서버 파이프라인으로 재구성한 것입니다. 실제 오류는 밝기, Tesseract의 페이지 분할, 상호명 줄바꿈, 결제 금액과 잔액의 혼동이 함께 만든 복합 문제였습니다.

현재 판단: 외부 OCR 서비스 도입 전 단계로서 충분히 의미 있는 개선입니다. 다만 표본이 4장뿐이므로 “범용적으로 100% 정확하다”는 의미는 아닙니다. 이후에는 실제 실패 이미지를 익명화된 회귀 표본으로 축적하며 규칙을 일반화해야 합니다.

2. 오늘 작업 범위와 의사결정 흐름

  1. 초기 문제 확인

    어두운 거래 화면과 복잡한 목록 UI에서 인식 품질 저하

    상태바, 아이콘, 줄바꿈된 상호명과 금액이 한 OCR 텍스트에 섞였고 일부 한글은 라틴 문자로 반복 오인식됐습니다.

  2. 아키텍처 결정

    프론트엔드 Tesseract.js에서 백엔드 Native Tesseract로 이전

    브라우저별 성능 차이와 모델 다운로드 부담을 제거하고, 동일한 모델·전처리·파서를 서버에서 재현하도록 했습니다. 외부 OCR은 비용과 개인정보 전송 검토 전까지 보류했습니다.

  3. 기본 파이프라인 구축

    원본·대비 강화·반전·이진화 변형을 생성하고 결과 선택

    한 번의 전처리 결과에 의존하지 않고 서로 다른 영상 변형을 인식해 거래 후보 수와 상호명 품질을 기준으로 결합했습니다.

  4. 사용 흐름 개선

    다중 이미지 업로드와 서버 분석으로 UI 변경

    최대 10장을 선택할 수 있게 하고, 브라우저에서는 추출 원문을 노출하지 않고 사용자가 확인해야 하는 거래 후보만 보여주도록 했습니다.

  5. 4번 이미지 추가 분석

    누락 1건과 잔액·상태바 오탐 9건을 분리 진단

    PSM 6과 PSM 11 원문을 비교해 아성다이소 누락은 페이지 분할 문제, 잔액 오탐은 파서의 의미 구분 문제임을 확인했습니다.

  6. 일반화와 회귀 검증

    희소 OCR 조건부 실행, 상태 기반 파싱, 순서 기반 병합

    새 상호명을 추가로 하드코딩하지 않고 화면 구조를 인식하는 규칙을 적용했으며, 1~4번 실제 이미지 24건을 다시 확인했습니다.

4. 발견한 문제

인식 단계의 문제

  • 다크 모드에서 글자와 배경 대비가 부족하거나 반전 방향이 맞지 않음
  • 목록형 2열 UI를 PSM 6이 하나의 텍스트 블록으로 잘못 가정
  • 화면 끝의 아성다이소 -8,200원 누락
  • 굵은 한글 긱스타PC를 여러 변형에서 ZIAEIPC 계열로 동일하게 오인식
  • 긴 법인명과 코엑스몰 같은 한 글자 접미사가 여러 줄·스크롤 흔적으로 분리

해석 단계의 문제

  • 결제 금액과 결제 후 잔액이 모두 원화 숫자라 형태만으로 구분 불가
  • PSM 11에서 상호와 금액이 서로 다른 줄로 분리됨
  • 상태바의 시간·배터리 숫자가 금액 후보에 섞일 수 있음
  • 서로 다른 OCR 결과를 단순 합치면 중복 거래 발생
  • 단순 중복 제거는 실제 연속 동일 결제를 삭제할 수 있음
  • 날짜 영역이 잘린 거래는 실제 날짜를 복원할 근거가 없음

5. 원인 진단 방법

OCR 문제를 한 덩어리로 보지 않고 영상 전처리 → 문자 인식 → 텍스트 파싱 → 후보 병합 단계로 나눴습니다. 각 단계의 출력을 사람 판독 기준값과 대조해 오류가 처음 발생한 지점을 찾았습니다.

  1. Ground truth 작성: 이미지에서 사람이 읽은 상호명·금액·날짜를 정답 목록으로 고정했습니다.
  2. OCR 원문 비교: 원본·강화·이진화 이미지와 PSM 6·11의 텍스트를 각각 확인했습니다.
  3. 누락 위치 추적: OCR 원문에 없는지, 원문에는 있지만 파서가 버렸는지 구분했습니다.
  4. 오탐 패턴 분류: 잘못 생성된 숫자가 상태바인지, 잔액인지, 날짜인지 원본 UI와 대조했습니다.
  5. 가설별 최소 수정: 밝기 문제에는 전처리, 레이아웃 문제에는 PSM, 의미 문제에는 파서 규칙을 적용했습니다.
  6. 회귀 검증: 새 이미지뿐 아니라 기존 1~3번을 같은 실제 API 경로로 다시 실행했습니다.
중요한 관찰: 4번은 밝은 화면이므로 전처리만 강화해도 해결되지 않았습니다. 아성다이소 누락의 직접 원인은 페이지 분할 모드였고, 9건의 오탐은 OCR보다 거래 파서의 문맥 이해 부족이 원인이었습니다.

6. 구현한 개선

6.1 서버 Native Tesseract와 모델 고정

개발 Docker 이미지에 Tesseract 5를 설치하고 tessdata_best의 한국어·영어 모델을 특정 commit으로 고정했습니다. 다운로드된 모델의 SHA-256을 확인하므로 같은 코드가 다른 모델 파일을 우연히 사용하는 위험을 줄였습니다. 실행 언어는 kor+eng, 기본 timeout은 인식 1회당 20초입니다.

6.2 전처리 변형 생성

  • Original: 투명도와 색상 형식을 RGB로 정규화합니다.
  • Enhanced: 회색조 히스토그램의 2~98 percentile을 0~255로 늘려 대비를 강화합니다.
  • Dark-mode inversion: 평균 휘도가 128 미만이면 밝고 어두운 방향을 반전합니다.
  • Binary: Otsu 임계값으로 흑백 이진화합니다. 기본 결과가 비거나 두 기본 변형의 후보 수가 다를 때만 추가 인식합니다.

6.3 조건부 PSM 11

기본 PSM 6 결과에 HH:mm #결제수단 형태가 있으면 카드 목록형 화면으로 판단해 원본 이미지에 PSM 11을 추가 실행합니다. PSM 11은 흩어진 텍스트를 독립적으로 탐색하므로 화면 하단의 누락 거래를 복원하는 데 유리합니다. 모든 이미지에 실행하지 않아 일반 입력의 불필요한 지연을 제한했습니다.

6.4 상태 기반 거래 파서

파서는 줄 하나만 보는 정규식 모음이 아니라, 이전 줄의 의미를 기억하는 작은 상태 머신으로 동작합니다.

상호명              → leadingMerchant에 임시 보관
단독 결제 금액       → 직전 상호와 묶어 거래 후보 생성
HH:mm #결제수단     → 현재 후보 확정, 다음 단독 금액 억제
다음 단독 금액      → 잔액으로 판단하고 제외
날짜 헤더            → 이후 거래의 currentDate 갱신

이 구조로 같은 숫자 형식이어도 위치와 앞뒤 행에 따라 결제 금액과 잔액을 구분합니다. 상호가 없는 상태의 단독 숫자는 버려 상태바 숫자가 다음 거래에 붙는 문제도 차단했습니다.

6.5 후보 선택과 순서 기반 병합

각 OCR 시도의 결과는 거래 후보 수를 가장 크게 보고, 한글·문자 비율이 높고 숫자 잡음이 적은 상호명에 가점을 주어 평가합니다. 결과를 결합할 때는 금액의 등장 순서로 LCS를 구해 같은 거래를 대응시킵니다. 대응된 거래에서는 더 깨끗한 상호명을 선택하고, 다른 시도에만 있는 타당한 거래를 삽입합니다.

병합 후 단순 중복 제거를 하지 않는 것이 중요합니다. 예를 들어 같은 카페에서 5,000원을 연속 결제한 두 건은 데이터만 보면 중복처럼 보이지만 실제로는 서로 다른 거래일 수 있습니다.

6.6 좁은 예외 보정

긱스타PC는 원본·강화·이진화 모두에서 같은 라틴 문자 형태로 오인식돼 영상 변형만으로 복구할 정보가 없었습니다. 따라서 ZIAEIPC, 2IAELPC 등 관측된 소수 변형만 긱스타PC로 바꾸는 좁은 보정이 남아 있습니다. 줄 끝의 코엑스몰이 스크롤 흔적과 함께 분리되는 경우에도 원문 줄 구조를 확인하는 제한적 접미사 복원 규칙이 있습니다.

이 규칙들은 현재 표본에는 효과적이지만 범용 OCR 지식은 아닙니다. 새로운 상호마다 같은 방식으로 하드코딩하면 유지보수 비용과 잘못된 자동 교정 위험이 빠르게 증가합니다.

6.7 API와 화면

  • POST /api/ledgers/{ledgerId}/transaction-imports/ocr-preview 추가
  • 반복 image multipart 필드로 최대 10장 수신
  • 첨부 순서대로 처리하고 여러 이미지의 candidate ID에 이미지 순번 prefix 부여
  • 인증과 장부 멤버 권한 확인 후 OCR 실행
  • 브라우저의 Tesseract.js 의존성 제거
  • 프론트에서는 추출 원문을 렌더링하지 않고 거래 후보만 표시
  • 직접 입력 텍스트 preview 기능은 별도 경로로 유지

API 응답에는 진단과 호환을 위해 아직 extractedText가 포함되지만, 사용자 화면에서는 보여주지 않습니다. 이는 “UI에서 숨김”이지 “서버 응답 계약에서 제거”는 아닙니다.

7. 이번 작업에 사용된 기술 지식

OCR은 하나의 모델 호출이 아니라 파이프라인이다

OCR의 최종 정확도는 문자 모델만으로 결정되지 않습니다. 입력 영상의 대비, 페이지 레이아웃 가정, 언어 모델, 행 순서 복원, 도메인 규칙이 연쇄적으로 영향을 줍니다. 따라서 오류를 고칠 때도 “Tesseract가 틀렸다”가 아니라 어느 단계에서 정보가 사라졌는지를 먼저 확인해야 합니다.

휘도와 대비 스트레칭

RGB 픽셀은 사람의 밝기 인지에 가까운 가중치 0.299R + 0.587G + 0.114B로 회색조 휘도로 변환합니다. 최솟값과 최댓값을 그대로 쓰면 한두 개의 극단 픽셀에 영향을 받으므로, 하위 2%와 상위 2%를 잘라낸 percentile 범위를 0~255로 확장합니다. 평균 휘도가 낮은 이미지는 반전해 검은 글자·흰 배경에 가까운 형태를 만듭니다.

Otsu 이진화

Otsu 방식은 회색조 히스토그램을 배경과 전경 두 집단으로 나눴을 때 집단 사이 분산이 가장 커지는 임계값을 찾습니다. 수동으로 “밝기 128 이상” 같은 값을 정하지 않아도 이미지별 임계값을 얻을 수 있습니다. 다만 그림자나 그라데이션이 심한 경우에는 전역 임계값 하나로 충분하지 않을 수 있습니다.

모드가정이번 작업에서의 역할
PSM 6이미지를 하나의 균일한 텍스트 블록으로 간주행 흐름이 비교적 일정한 화면의 기본 인식
PSM 11희소한 텍스트를 가능한 한 독립적으로 탐색2열 목록과 화면 끝에 흩어진 거래 복구

PSM 11은 누락을 줄일 수 있지만 행 관계를 더 많이 잃으므로 파서가 상호·금액·메타데이터를 다시 연결해야 합니다.

LCS를 이용한 결과 융합

LCS는 두 순서열에서 순서를 유지하며 공통으로 나타나는 가장 긴 부분 수열을 찾습니다. 이번 구현에서는 금액 열을 비교 키로 사용해 서로 다른 OCR 시도의 거래 순서를 맞춥니다. 전체 줄 문자열을 동일 비교하지 않기 때문에 상호명이 조금 다르게 인식돼도 같은 금액·순서의 거래를 대응시킬 수 있습니다.

정밀도와 재현율

  • 오탐(False positive): 잔액이나 상태바 숫자를 거래라고 판단하는 경우
  • 누락(False negative): 아성다이소처럼 실제 거래를 찾지 못한 경우
  • 정밀도(Precision): 생성한 후보 중 실제 거래의 비율
  • 재현율(Recall): 실제 거래 중 찾아낸 비율

후보 수만 늘리는 것은 재현율을 높일 수 있지만 오탐도 늘립니다. 이번 개선은 PSM 11로 누락을 복구하면서 문맥 파싱으로 잔액 오탐을 제거해 두 방향을 함께 개선한 사례입니다.

현재 confidence 값의 의미

응답의 confidence는 현재 Tesseract가 반환한 글자별 신뢰도가 아닙니다. 카테고리가 매칭되면 0.82, 그렇지 않으면 0.68을 주는 애플리케이션 수준 값입니다. 향후 낮은 OCR 신뢰도만 사용자에게 확인시키려면 TSV/hOCR 등의 실제 Tesseract confidence를 별도로 수집해야 합니다.

9. 한계와 추후 예상 문제

우선도예상 문제왜 발생하는가대응 방향
높음 새 금융 앱에서 잔액·할인·포인트를 거래로 오인 현재 문맥 규칙은 시간 + #결제수단 + 잔액 구조에 최적화 레이아웃별 익명화 표본과 negative fixture 추가
높음 여러 이미지 요청의 긴 지연과 CPU 부하 이미지를 첨부 순서대로 처리하고 이미지마다 여러 번 Tesseract 실행 동시성 제한, 작업 큐, 이미지별 제한 병렬화, timeout 관측
중간 날짜가 잘린 상단 거래의 잘못된 기본 날짜 이미지에 정보가 없으면 OCR로 복구 불가능 추정 표시, 사용자 확인, 주변 이미지와의 날짜 연결 검토
중간 동일 금액이 반복될 때 LCS 대응 오류 현재 병합의 주 비교 키가 금액과 순서 날짜·상호 유사도·행 좌표를 결합한 정렬 검토
중간 하드코딩 교정 규칙 증가 모든 변형이 같은 오인식을 내면 현재 정보만으로 원문 복구 불가 실제 OCR confidence, 사용자 상호 사전, 확인 UI를 제한적으로 결합
중간 그라데이션·그림자·기울어진 촬영 이미지 현재 Otsu는 이미지 전체에 하나의 임계값을 적용하고 기울기 보정이 없음 adaptive threshold, deskew, 영역 감지의 필요성을 표본으로 평가
관찰 컨테이너 밖 실행 시 OCR 불가 호스트에 Tesseract와 kor/eng 모델이 없을 수 있음 환경 문서 준수와 시작 시 dependency health check 검토
관찰 실제 OCR 신뢰도 부재 현재 API confidence는 카테고리 매칭 점수 낮은 신뢰도 항목 확인 기능 도입 전에 TSV/hOCR 데이터 수집

10. 앞으로의 표본 확장과 개선 원칙

당분간은 외부 OCR로 바로 교체하기보다 현재 엔진의 실패 표본을 체계적으로 모으는 방향이 적절합니다. 표본 한 장을 추가할 때는 성공 이미지만 저장하는 것이 아니라 정답과 실패 이유를 함께 기록해야 합니다.

표본 하나에 남길 정보

  • 익명화된 원본 또는 재현 가능한 테스트 이미지
  • 사람이 판독한 상호명·금액·날짜의 ground truth
  • 원본·강화·이진화 및 PSM별 OCR 결과
  • 오류 유형: 문자 오인식, 행 누락, 잔액 오탐, 날짜 오류, 병합 오류
  • 수정 전후 후보와 처리 시간
  • 새 규칙이 기존 표본을 깨뜨리지 않는 회귀 테스트

우선 수집할 커버리지

축필요한 표본
화면라이트·다크 모드, 긴 목록, 카드형 UI, 좌우 2열, 팝업이 겹친 화면
금액 의미결제, 잔액, 취소, 환불, 승인, 할인, 포인트, 할부
텍스트긴 법인명, 영문 로고, 숫자가 포함된 상호, 같은 상호·금액 반복
날짜전체 날짜, 월·일 헤더, 연도 경계, 날짜가 잘린 이미지
화질저해상도, 압축, 흔들림, 기울기, 그림자, 스크린 촬영
업로드1장, 10장, 합계 10MB 근접, 잘못된 MIME, 초대형 해상도

개선 여부를 결정하는 기준

  1. 새 실패가 영상 문제인지, 인식 문제인지, 의미 파싱 문제인지 먼저 분류합니다.
  2. 특정 상호명 치환보다 화면 구조·문맥에 적용되는 일반 규칙을 우선합니다.
  3. 규칙을 추가하기 전에 오탐을 방지하는 반례 테스트도 함께 만듭니다.
  4. 정확도뿐 아니라 이미지당 시간과 동시 요청 CPU를 같이 기록합니다.
  5. 표본이 충분해진 뒤 현재 엔진과 외부 OCR을 같은 ground truth로 비교합니다.
외부 OCR 재평가 시점: 표본이 늘어나도 현재 Tesseract의 문자 누락·오인식이 전처리와 파싱으로 해결되지 않거나, 처리 시간과 운영 복잡도가 수용 기준을 넘을 때입니다. 이때 정확도, 응답 시간, 비용, 개인정보 외부 전송, credential 운영을 동일한 평가표에서 비교해야 합니다.

12. 결론

오늘 개선으로 OCR 실행 환경은 브라우저에서 서버로 일원화됐고, 어두운 화면과 2열 거래 목록을 위한 다중 전처리·다중 PSM 경로가 생겼습니다. 거래 파서는 숫자 모양만 보는 방식에서 결제 메타데이터와 행 순서를 이해하는 방식으로 발전했습니다. 현재 4개 이미지 24건은 모두 정답과 일치하지만, 다음 단계의 품질은 더 많은 표본을 얼마나 구조적으로 축적하느냐에 달려 있습니다.

따라서 앞으로의 원칙은 실패 표본 추가 → 오류 단계 분류 → 일반화 가능한 최소 수정 → 전체 회귀 검증입니다. 이 과정에서 더 이상 로컬 규칙으로 해결하기 어려운 정확도·성능 한계가 관측될 때 외부 OCR 서비스를 비교하는 것이 합리적입니다.

이 보고서는 2026-07-21 우리로그 로컬 개발 작업 트리와 실제 1~4번 이미지 검증 결과를 기준으로 작성되었습니다. 수치와 동작은 이후 구현 변경 및 회귀 표본 확대에 따라 갱신될 수 있습니다.
